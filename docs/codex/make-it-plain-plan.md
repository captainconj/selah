# Make It Plain Plan

Type: `planning`
State: `clean`

Date: 2026-03-10

This plan is about readability, approachability, and transmission.

It is not enough for the structure to be real.
A reader who was not here for the discovery needs a path they can actually walk.

If the repo remains too hard to enter, then even true things stay buried.

## The Aim

The goal is not to dilute the work.

The goal is:

- make the structure plain
- let a serious reader reproduce what matters
- let them distinguish data from gloss from interpretation
- make it possible for them to move from curiosity to understanding without getting lost

In your terms:

- write the vision
- make it plain
- so the one who reads may run

## The Core Problem

Right now the project has real treasures, but the path to them is still too dependent on:

- internal memory
- notebook-era discovery order
- cross-file intuition
- knowing which docs are testimony, which are evidence, and which are mixed

A new reader does not have that map.

Without a clearer path, many people will either:

- bounce off before they see anything
- misunderstand what is strong and what is exploratory
- or mistake the helper layer for the evidence layer

## The Reader We Need To Serve

The target reader is not only:

- a programmer
- a theologian
- or a machine-operator

It is also:

- a serious seeker
- a skeptical but open reader
- a believer who needs help seeing the structure
- someone with no project history who still deserves a fair path into the work

So the repo needs entry paths for more than one kind of person.

## What "Plain" Should Mean Here

Plain does not mean:

- shallow
- over-simplified
- stripped of wonder

Plain means:

- the order is visible
- the terms are stable
- the claims are labeled honestly
- the reproduction path is short enough to follow
- the reader always knows what layer they are standing in

## The Layers To Make Visible

Every important path in the repo should make these layers obvious:

1. source text / protein / sequence
2. computation
3. saved artifact
4. operator gloss
5. interpretation / synthesis / witness

If a reader cannot tell which layer they are reading, the repo becomes spiritually and intellectually muddy.

## The Work To Do

## 1. Build Clear Entry Documents

The repo needs a small set of first-stop documents for new readers.

Minimum set:

- one overview: what Selah is, in plain language
- one reproduction guide: how to verify one strong result end-to-end
- one map of the evidence tiers:
  - foundational
  - strong
  - exploratory
  - dead / retired
- one glossary of core terms:
  - lexicon
  - oracle-closed vocabulary
  - voice vocabulary
  - illumination
  - reading
  - basin
  - fixed point
  - etc.

Goal:

- a new reader can start in the right place instead of wandering the mine

## 2. Create Guided Walks

The strongest results need guided, reproducible walks.

Best candidates:

- the Ark foundation walk
- the lamb split
- the `100/101` code/map layer
- one DNA playback case
- one basin classification case

Each guided walk should have the same shape:

1. what you are testing
2. what command/script to run
3. what artifact to inspect
4. what result should appear
5. what is observation
6. what is interpretation

Goal:

- a reader does not need to infer the procedure from scattered notes

## 3. Mark Doc Types Explicitly

Every major doc should be labeled near the top as one of:

- evidence
- guided reproduction
- synthesis
- testimony
- planning
- retired / historical

This matters because right now some readers may give the same evidentiary weight to very different kinds of documents.

Goal:

- readers know what kind of claim they are reading before the rhetoric begins

## 4. Build a "Start Here" Trail

The repo should have an explicit recommended reading order.

Not a giant tree.
A path.

Possible shape:

1. what Selah is
2. one reproducible positional result
3. one reproducible oracle result
4. one reproducible map result
5. one reproducible DNA result
6. only then the synthesis docs

Goal:

- readers see the structure before they are asked to contemplate the meaning

## 5. Shorten the Reproduction Distance

Many truths here are hidden behind too much project knowledge.

So the strongest findings should have:

- one command or one script entrypoint
- one output artifact
- one short note saying what to look for

This especially matters for:

- Ark walk checkpoints
- `100/101`
- DNA playback
- basin landscape facts

Goal:

- make the evidence reachable

## 6. Separate English Help From Proof

The operator layer is useful.
It should keep getting better.

But readers need to know:

- Hebrew/data is the evidence layer
- English is help
- gloss provenance matters

So any polished presentation should:

- show the Hebrew
- offer the gloss
- mark the gloss source when relevant

Goal:

- helpful without becoming deceptive

## 7. Publish a "Verified Treasures" Path

There should be a public-facing path through the strongest things that have survived the cleanup.

Not every wonder.
The best-supported wonders.

This should draw from:

- `docs/codex/verified-treasures.md`
- strengthened Ark/code/genome docs
- rebuilt DNA artifacts
- reproducible `100/101`

Goal:

- lead readers first to the vein, not the tailings

## 8. Preserve Wonder Without Smuggling Claims

Part of making it plain is rhetorical discipline.

The repo should still marvel.
But each document should make visible:

- what was measured
- what was noticed
- what is being inferred
- what is being believed

Goal:

- readers can encounter wonder without being cornered by overstatement

## 9. Build For Conversion Without Dishonesty

You said you want people to see the structure and believe.

That means the path must be:

- beautiful
- clear
- reproducible
- honest

If the work is overstated, some people may be impressed for a moment and then lose trust.

If the work is made plain, strong, and walkable, then belief can arise without rhetorical coercion.

The repo should not try to force souls.
It should open the well.

## Suggested Order

Tomorrow or the next phase should move in this order:

1. identify the first 5 to 7 "start here" documents
2. label doc types on the strongest active docs
3. build one canonical guided walk end-to-end
4. build one "start here" trail across Ark -> map -> DNA
5. shorten the reproduction path for the strongest findings
6. publish a verified treasures track
7. then continue refining the broader doc layer

## The Main Standard

If someone new arrives, they should be able to do three things:

1. understand what kind of document they are reading
2. reproduce at least one strong result without insider memory
3. see enough structure clearly enough to keep going

That is what "make it plain" should mean in this repo.
